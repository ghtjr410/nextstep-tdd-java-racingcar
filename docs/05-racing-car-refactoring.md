## 나의 학습 목표
### 1. TDD 사이클을 끝까지 지킨다.
- 구현보다 테스트를 먼저 작성하고
- 작은 실패 -> 작은 성공 -> 리팩토링의 과정을 일관되게 유지한다.
- 테스트 없이 구현하는 습관을 절대 허용하지 않는다.
- 또한 red -> green -> refactor 단위로 커밋한다.
### 2. 객체지향 생활 체조 원칙을 의식적으로 지킨다.
### 3. 속도보다 사고의 깊이를 우선한다.
- 미션을 빠르게 끝내기 위한 타협을 하지않고
- "왜?" 라는 질문을 통해 설계 의도를 명확히 한다.
- 스스로 현재 구현의 의도를 100% 납득하고 설득할 수 있을 때까지 고민해본다.
### 4. 이전 단계의 피드백을 모두 반영한다.
- 1~4단계에서 받은 피드백을 리스트업하여 체크리스트로 관리한다.
- 같은 피드백을 반복하지 않도록, 모든 구현은 사전·사후 체크리스트 점검을 필수로 한다.
### 5. 테스트 범위의 경계를 객체 책임 관점에서 설정한다.
- 무엇을 테스트해야 하고, 무엇은 테스트하지 않아야 하는지를 각 객체의 책임과 협력 구조를 기준으로 판단한다.
- 상태를 검증하는 테스트를 중심으로 구성한다.
### 6. 주생성자와 부생성자를 적극적으로 활용해본다.
- 주생성자는 객체가 반드시 가져야 하는 핵심 상태만 받도록 설계한다.
- 부생성자는 보조적 편의만 제공하도록 제한한다.
## 미션 요구사항
### 1. 지금까지의 자동차 경주 코드 전체 삭제
- 기존 구조나 패턴에 의존하지 않고 완전히 새롭게 출발한다.
### 2. 기능 목록 작성
- 기능을 구현 단위로 나누고
- "누가 어떤 책임을 가져야 하는가"를 중심으로 기능을 해체한다.
### 3. 아무것도 의존하지 않는 가장 작은 단위부터 모델링

## PR 전 자기 점검 체크리스트
- 객체지향 생활체조 원칙을 준수하였는가? 
- TDD 사이클로 구현하였는가?
- TDD 사이클 단위로 커밋하였는가?
- 객체에게 상태를 노출시키지 않고 책임을 맡겼는가?
- 테스트는 객체의 책임과 결과만 검증했는가?
- 불변성을 최대한 유지했는가?
- 도메인 용어로 네이밍했는가?
- 협력 구조가 자연스럽게 읽히는가?
